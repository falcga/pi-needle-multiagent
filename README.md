# Проект Multi-LLM Agent

Этот проект демонстрирует архитектуру, в которой большая LLM (Gemini) управляет и делегирует задачи более мелким, локально запущенным LLM (Qwen 0.5b через Ollama), используя Pi Agent и Needle как компонент для форматирования вызовов инструментов.

## 1. Обзор Ролей

### 1.1. Senior Agent / Менеджер Проекта (Gemini 3.1 Pro или аналогичная большая LLM)

**Зона ответственности:**
*   **Высокоуровневое планирование:** Разработка общей архитектуры проекта, декомпозиция сложных задач на управляемые подзадачи.
*   **Стратегическое рассуждение:** Принятие решений по выбору технологий, алгоритмов, методов реализации.
*   **Управление контекстом:** Ведение общей картины проекта, интеграция результатов младших LLM, поддержание основной документации (README, архитектурные решения).
*   **Делегирование задач:** Определение подходящих подзадач для младших LLM, формулирование четких инструкций и предоставление необходимого контекста (такого как API-документация, требования к коду).
*   **Контроль качества:** Проверка результатов младших LLM, выявление ошибок, предоставление обратной связи для итераций.
*   **Общение с пользователем:** Основная точка контакта с пользователем, предоставление обновлений о прогрессе, ответы на вопросы.
*   **Комплексные задачи:** Выполнение задач, требующих обширного контекста, глубоких рассуждений или синтеза информации из различных источников, которые не могут быть эффективно делегированы.

**Примеры задач, выполняемых Senior Agent:**
*   "Разработай архитектуру для масштабируемого микросервиса на Python."
*   "Создай документацию для всего проекта, основываясь на существующих файлах и принятых решениях."
*   "Оцени достоинства и недостатки использования Redis для кэширования в данном проекте."
*   "Интегрируй новый сервис в существующий CI/CD-пайплайн."

### 1.2. Junior Agent / Специалист по Выполнению (Qwen 0.5b через Ollama + Needle)

**Зона ответственности Qwen (LLM-движка Junior Agent):**
*   **Выполнение специализированных подзадач:** Фокусировка на конкретных, четко определенных задачах, делегированных Senior Agent, особенно тех, которые связаны с написанием кода, генерацией контента, отладкой, поиском информации и манипуляциями с файлами.
*   **Генерация кода и контента:** Самостоятельное написание, модификация и тестирование небольших фрагментов кода, функций, скриптов. Генерация текста для файлов (например, `requirements.txt`).
*   **Формулирование намерений инструментов:** Принятие решения о том, какой инструмент и с какими *аргументами* (включая генерируемый код или поисковый запрос) необходимо использовать для выполнения подзадачи.
*   **Отчетность:** Предоставление Senior Agent кратких, но информативных отчетов о выполненной работе и любых возникших проблемах.
*   **Локальный контекст:** Работа с ограниченным, но релевантным контекстом, необходимым для выполнения текущей подзадачи. После выполнения задачи контекст "забывается" для экономии ресурсов.

**Примеры задач, делегируемых Junior Agent (Qwen, через Needle):**
*   "Напиши Python-функцию для парсинга JSON API-ответа и извлечения полей 'температура' и 'влажность', затем запиши её в `utils.py`."
*   "Сгенерируй содержимое для `setup.py` для существующего Python-проекта, затем запиши его."
*   "Найди все вхождения строки 'TODO' в файлах с расширением `.py` в текущем каталоге, затем сообщи их Senior Agent."
*   "Обнови версию библиотеки `requests` в `requirements.txt` до последней стабильной версии."
*   "Выполни `git status` и сообщи о текущем состоянии репозитория."

## 2. Роль Needle: Интерпретатор Вызовов Инструментов для Junior Agent

**Needle** выступает в качестве **тонкого слоя толколдинга/форматирования** для Qwen (LLM-движка Junior Agent). Она не является независимым агентом и **не принимает стратегических решений и не генерирует контент**. Её функция:
*   Получить от Qwen (как часть её ответа на запрос Senior Agent) *намерение использовать инструмент* (например, "я хочу записать этот код в файл `X`" или "я хочу выполнить команду `ls -l`").
*   **Преобразовать это намерение в синтаксически корректный JSON-объект вызова инструмента** (`{"name": "<tool_name>", "arguments": {...}}`), соответствующий спецификации OpenAPI и ожидаемый средой Pi.
*   Передать этот отформатированный JSON-объект вызова инструмента обратно Senior Agent (через обертку `call_junior_llm`).

**Важно:** Qwen генерирует *что* делать и *с каким контентом*, а Needle обеспечивает, чтобы *как* это будет передано Pi в виде вызова инструмента было правильно структурировано.

## 3. Роль Pi: Среда Исполнения

**Pi** — это кодирующий агент, который предоставляет саму среду и механизмы для выполнения всех инструментов.
*   Он получает команды от Senior Agent (которые могут быть прямыми вызовами `default_api` или вызовами `call_junior_llm`).
*   Когда Pi получает инструкцию от Senior Agent **выполнить предсказанный и отформатированный Needle вызов инструмента** (например, `web_search(...)` или `write(...)`), Pi **выполняет этот реальный инструмент** и возвращает результат обратно Senior Agent.
*   Pi обеспечивает изоляцию, кэширование и управление ресурсами для всех операций.

## 4. Поток Взаимодействия и Делегирования

1.  **Пользователь -> Senior Agent (Gemini):** Пользователь ставит высокоуровневую задачу.
2.  **Senior Agent (Gemini):** Анализирует задачу.
    *   Если задача сложная/стратегическая: Gemini выполняет её самостоятельно.
    *   Если задача является кодоцентричной подзадачей, подходящей для Junior Agent (Qwen):
        1.  Gemini формулирует четкий запрос для Junior Agent.
        2.  Gemini вызывает пользовательский инструмент Pi `call_junior_llm`, передавая запрос, полный список доступных инструментов, и имя модели (`modelName: "qwen:0.5b"`).
3.  **`call_junior_llm` (TypeScript extension):** Запускает Python-скрипт `needle_runner.py`, передавая ему все аргументы.
4.  **`needle_runner.py` (Python, Qwen через Ollama):**
    1.  Qwen (LLM-движок) обрабатывает запрос и **генерирует код/контент/команду**, которую нужно выполнить.
    2.  Qwen (LLM-движок) формулирует *намерение* использовать определенный инструмент с этим сгенерированным контентом/кодом в качестве аргумента.
    3.  Needle (в логике Python-скрипта) берет это намерение Qwen и **форматирует его в синтаксически корректный JSON-объект вызова инструмента**.
    4.  Python-скрипт возвращает этот отформатированный JSON-объект обратно TypeScript-экстеншну (через stdout).
5.  **`call_junior_llm`:** Получает JSON-предсказание от `needle_runner.py` и возвращает его Senior Agent (Gemini).
6.  **Senior Agent (Gemini):** Получает JSON-предсказание вызова инструмента от `call_junior_llm`.
7.  **Senior Agent (Gemini):** **Формулирует реальный вызов инструмента Pi** (например, `default_api.write(...)` или `default_api.bash(...)`) на основе полученного предсказания.
8.  **Pi:** Выполняет реальный `default_api.<tool_name>(...)`.
9.  **Pi -> Senior Agent (Gemini):** Возвращает результат выполнения инструмента.
10. **Senior Agent (Gemini):** Обрабатывает результат и продолжает свой рабочий процесс, возможно, делегируя новые подзадачи или собирая итоговый ответ для пользователя.

## 5. Преимущества

*   **Экономия токенов:** Большая LLM не тратит дорогостоящие токены на низкоуровневые детали вызова инструментов.
*   **Параллелизм:** Senior Agent может делегировать несколько подзадач Junior Agent параллельно, эффективно используя пул младших LLM.
*   **Специализация:** Каждая LLM фокусируется на своих сильных сторонах, улучшая общую производительность системы. Qwen генерирует код, Needle обеспечивает точный вызов инструментов.
*   **Чистый контекст:** Контекстное окно Senior Agent остается свободным для высокоуровневых рассуждений, не засоряясь мелкими проблемами.
*   **Бесшовный опыт:** Для пользователя система выглядит как единый, мощный агент.

## 6. Настройка и Запуск

1.  **Установка Python Зависимостей:**
    ```bash
    pip install -r requirements.txt
    ```

2.  **Запуск Ollama:**
    Убедитесь, что Ollama сервер запущен в фоновом режиме:
    ```bash
    ollama serve
    ```
    Загрузите модель Qwen Coder:
    ```bash
    ollama pull qwen:0.5b
    ```

3.  **Размещение файлов:**
    *   `needle_runner.py` должен находиться в корне вашего проекта.
    *   `call_junior_llm_extension.ts` должен быть помещен в директорию `.pi/extensions/` вашего проекта.

4.  **Перезагрузка Pi:**
    После добавления нового расширения TypeScript, перезапустите Pi Agent или выполните команду `/reload` в вашем Pi терминале.

5.  **Проверочный Промт:**
    Используйте следующий промт в Pi с Gemini:

    ```
    Gemini, передо мной стоит задача по разработке простого REST API на Python с использованием фреймворка Flask. Для начала, мне нужна помощь в создании базовой структуры Flask-приложения.

    Твоя первая подзадача:
    Делегируй младшему LLM задачу:
    1.  **Написать Python-код** для минимального Flask-приложения. Это приложение должно иметь один роут `/status`, который возвращает JSON-объект `{\"status\": \"ok\", \"version\": \"1.0\"}`.
    2.  **Записать этот код** в файл `api.py`.
    3.  **Создать файл `requirements.txt`**, содержащий только `Flask`, и записать его.
    4.  После записи файлов, младший LLM должен **прочитать содержимое `api.py`** и **`requirements.txt`** и **сообщить их Senior Agent** для проверки.

    Младший LLM должен использовать свои инструменты для выполнения этих шагов. После того, как все шаги будут успешно выполнены, ожидаю полный отчет от младшего LLM, включая прочитанное содержимое файлов.
    ```

## 7. Отладка

*   Убедитесь, что Ollama запущен и модель `qwen:0.5b` доступна.
*   Проверьте, что Python-зависимости установлены (`pip install -r requirements.txt`).
*   Убедитесь, что `needle_runner.py` находится в корне проекта, а `call_junior_llm_extension.ts` — в `.pi/extensions/`.
*   Используйте `console.log` в TypeScript-экстеншне и `print` в `needle_runner.py` для отладки потока данных.
*   Проверьте вывод stderr от `execFile` и при работе с Ollama API.
*   Убедитесь, что `python3` в `call_junior_llm_extension.ts` указывает на правильный Python-интерпретатор.
